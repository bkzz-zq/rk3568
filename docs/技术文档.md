## 1. 项目架构

### 1.1 整体架构（视频直连 + 检测分离 + 多线程流水线）

```

│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐         │
```

### 1.2 为什么采用这种架构

| 传统方案 | 本项目方案 |
|---------|-----------|
| RK3568 编码 JPEG + Base64 推送 | RK3568 只发 JSON（~1KB/帧） |
| PC 显示压缩后的低画质画面 | PC 直连 RTSP 原始画质 |
| RK3568 CPU 占用高（编码） | RK3568 CPU 占用低（VPU 解码 + NPU 推理） |
| 帧率 ~3 FPS | 帧率 ~40 FPS |

## 2. 多线程流水线

### 2.1 架构设计

核心优化：解码、检测、推送三阶段完全解耦，互不阻塞。

```
时间线示意：

线程1(解码):   [帧1][帧2][帧3][帧4][帧5][帧6][帧7][帧8]...  持续 ~25fps
                  ↓ 最新帧覆盖
线程2(检测):   [---行人+NPU---][跳过][跳过][---行人+NPU---]...  每3帧检测一次
               [------车牌CPU----------]......[------车牌CPU----]...  每9帧检测一次
                  ↓ 结果缓存
主线程(推送):  [推送][推送][推送][推送][推送][推送][推送]...     ~40fps 不阻塞
```

| 线程 | 职责 | 频率 | 阻塞？ |
|------|------|------|--------|
| 解码线程 | GStreamer+MPP 硬解码读帧 | ~25fps | 否 |
| 检测线程 | NPU行人 + CPU车牌并行 | 行人:每3帧, 车牌:每9帧 | 否 |
| 主线程 | 读缓存 → WebSocket 推送 | ~40fps | 否 |

### 2.2 数据流

```python
# main.py 核心结构

# 共享数据（线程安全）
results_lock = threading.Lock()
cached_results = {"persons": None, "plates": None, "texts": None}

# 线程2: 检测循环
def detection_loop():
    executor = ThreadPoolExecutor(max_workers=3)
    while running:
        frame = camera.get_frame()              # 取最新帧
        futures = {
            executor.submit(person_detector.detect, small_frame): "person",
            executor.submit(plate_detector.detect, frame): "plate",
        }
        for future in as_completed(futures):     # 并行等结果
            result = future.result()
            with results_lock:
                cached_results[name] = result     # 更新缓存

# 主线程: 推送循环
while running:
    with results_lock:
        persons, plates = cached_results["persons"], cached_results["plates"]
    pusher.push_results(persons, plates)         # 不等待检测完成
    time.sleep(0.02)                             # ~50fps 上限
```

### 2.3 帧传递方式

采用 **"最新帧"模式**（不用队列），避免帧积压：

```python
# CameraStream 内部
with self._lock:
    self._frame = frame    # 始终覆盖，只保留最新帧

def get_frame(self):
    with self._lock:
        return self._frame.copy()
```

### 2.4 为什么车牌不使用 NPU

RK3568 只有 **1 个 NPU**（0.8 TOPS），无法同时运行两个模型：

| 模型 | 后端 | 原因 |
|------|------|------|
| YOLOv8 行人检测 | NPU | 优先占用 NPU，每 3 帧一次 |
| 车牌检测 | CPU (HyperLPR3) | 避免与行人检测争抢 NPU |

如果两个模型都用 NPU，需要交替加载/卸载，严重拖慢推理。

## 3. GStreamer + MPP 硬解码

### 3.1 Pipeline 架构

```
rtspsrc → rtph264depay → h264parse → mppvideodec → videoconvert → videoscale → appsink
                                     ↑ VPU硬解码    ↑ BGR转换     ↑ 硬件缩放640x360
                                     CPU≈0          ~5ms          ~3ms
```

### 3.2 自动检测流程

```python
# camera.py 启动时自动检测

1. 检查 OpenCV 是否有 GStreamer 后端
   cv2.getBuildInformation() → 搜索 "GStreamer: YES"

2. 检查 mppvideodec 插件是否可用
   gst-inspect-1.0 mppvideodec

3. 都有 → GStreamer + MPP 硬解码 + videoscale 缩放
4. 缺一 → 回退 FFmpeg 软解码
```

### 3.3 解码模式对比

| 解码模式 | CPU 占用 | 延迟 | 输出分辨率 |
|----------|---------|------|-----------|
| GStreamer + MPP (硬解码) | ~5% | ~5ms | 640x360 BGR |
| FFmpeg (软解码) | ~60% | ~50ms | 1920x1080 BGR |

### 3.4 坐标映射

GStreamer 输出 640x360 帧，但 PC 客户端在 1920x1080 RTSP 视频上叠加检测框：

```python
# camera.py 提供 output_scale 属性
self._output_scale = 640 / 1920  # = 0.33

# main.py 中还原坐标
for r in result:
    r["bbox"] = [int(v / detect_scale) for v in r["bbox"]]
    # 例: [100, 50, 200, 150] → [303, 152, 606, 455] (还原到 1920x1080)
```

### 3.5 系统OpenCV链接（踩坑记录）

pip 安装的 `opencv-python` **没有 GStreamer 后端**，需要链接到系统 OpenCV：

```bash
# 卸载 pip 版本
pip uninstall opencv-python opencv-python-headless -y

# 链接系统 OpenCV（有 GStreamer 支持）
ln -sf /usr/lib/python3/dist-packages/cv2 .venv/lib/python3.10/site-packages/cv2
ln -sf /usr/lib/python3/dist-packages/cv2.cpython-310-aarch64-linux-gnu.so .venv/lib/python3.10/site-packages/

# 验证
python3 -c "import cv2; print(cv2.getBuildInformation())" | grep -i gstreamer
# 应显示: GStreamer: YES
```

### 3.6 NumPy 版本兼容

系统 OpenCV 绑定的 NumPy 版本可能与 pip 安装的不兼容：

```bash
# 必须降级到 1.x（系统 OpenCV 不支持 NumPy 2.x）
pip install "numpy<2"
```

## 4. RK3568 NPU 详细配置

### 4.1 硬件信息

| 项目 | 参数 |
|------|------|
| SoC | RK3568 |
| CPU | 4 核 ARM Cortex-A55 @ 2.0GHz |
| NPU | 0.8 TOPS (INT8) |
| VPU | 支持 H.264/H.265 硬解码 |
| RGA | 2D 图形加速（缩放/CSC） |
| RAM | 建议 2GB+ |
| 架构 | aarch64 (ARM64) |

### 4.2 NPU 软件栈版本（关键！版本必须匹配）

| 组件 | 版本 | 说明 |
|------|------|------|
| **rknn-toolkit2** | 2.3.2 | PC 端模型转换工具（x86_64） |
| **rknn-toolkit-lite2** | 2.3.2 | 板子端推理库（aarch64） |
| **librknnrt.so** | 2.3.2 | NPU 驱动运行时 |
| **RKNN 驱动** | 0.9.0 | 内核驱动（固件自带） |

> ⚠️ **版本必须完全一致！** 如果 rknn-toolkit2 用 2.3.2 转换模型，板子上的 librknnrt.so 也必须是 2.3.2。

### 4.3 NPU 工作原理

```
PC 端（x86_64）                          板子端（aarch64）
┌──────────────┐                        ┌──────────────┐
│ rknn-toolkit2│                        │rknn-toolkit  │
│              │                        │   -lite2     │
│ ONNX/PyTorch │    .rknn 文件           │              │
│    ↓         │ ──────────────────────► │ RKNNLite()  │
│ RKNN 模型     │       传输              │    ↓         │
│ (.rknn)      │                        │ NPU 推理      │
│              │                        │ (硬件加速)    │
└──────────────┘                        └──────────────┘
```

## 6. NPU 行人检测代码详解

### 6.1 核心代码（src/detector/person.py）

```python
from rknnlite.api import RKNNLite

class PersonDetector:
    def _detect_rknn_combined(self, frame):
        """一次 NPU 推理，同时提取行人和车辆"""
        img, ratio, (dw, dh) = letterbox(frame, self.input_size)
        img = cv2.cvtColor(img, cv2.COLOR_BGR2RGB)
        img = np.expand_dims(img, 0).astype(np.uint8)

        # 一次 NPU 推理
        outputs = self.model.inference(inputs=[img])

        # 从同一输出中分别提取行人和车辆
        persons = self._postprocess_yolo(outputs[0], frame.shape, ratio, dw, dh)
        vehicles = self._postprocess_vehicles(outputs[0], frame.shape, ratio, dw, dh)
        return persons, vehicles
```

### 6.2 关键注意事项

#### ⚠️ 输入格式必须正确

```python
# 错误！3 维输入会报错
img = np.array(img, dtype=np.uint8)  # shape: (640, 640, 3)

# 正确！4 维输入
img = np.expand_dims(img, 0).astype(np.uint8)  # shape: (1, 640, 640, 3)
```

#### ⚠️ 不要手动归一化

```python
# 错误！NPU 内部会做归一化
img = img.astype(np.float32) / 255.0

# 正确！直接传 uint8
img = np.expand_dims(img, 0).astype(np.uint8)
```

## 7. 车牌识别（HyperLPR3 + ROI）

### 7.1 当前方案

```python
from hyperlpr3 import LicensePlateCatcher

catcher = LicensePlateCatcher()
results = catcher(frame)
# 返回: [(号牌, 置信度, 类型, [x1,y1,x2,y2]), ...]
```

### 7.2 ROI 区域裁剪优化

```yaml
# config.yaml
plate_recognition:
  roi: [0.1, 0.4, 0.9, 0.95]  # [左, 上, 右, 下] 归一化坐标
```

ROI 裁剪的好处：
1. 减少搜索区域 → HyperLPR3 更快
2. 避免误检
3. 车牌通常在画面下半部分

### 7.3 车牌识别性能

| 输入尺寸 | 无车牌耗时 | 有车牌耗时 |
|---------|-----------|-----------|
| 1920x1080 | ~150ms | ~400ms |
| 640x360 (pipeline输出) | ~120ms | ~300ms |

### 7.4 NPU 加速方案（长期规划）

车牌检测可以用 NPU 加速（检测 ~10ms），但因 RK3568 只有单 NPU，当前与行人检测冲突。完整方案见 `docs/车牌检测NPU加速方案.md`。

## 9. 性能优化记录

### 9.1 优化历程

| 阶段 | 优化措施 | FPS | 关键变化 |
|------|---------|-----|---------|
| 初始 | 全帧检测 + JPEG 编码推送 | ~3 | 基线 |
| 第1轮 | 缩放到640px + 每3帧检测 | ~8 | 减少 CPU 计算 |
| 第2轮 | 车牌降频每9帧 + 去掉图像编码 | ~10 | 减少 HyperLPR3 调用 |
| 第3轮 | NPU 行人 + RTSP 直连分离架构 | ~10 | NPU 加速行人检测 |
| **第4轮** | **GStreamer+MPP 硬解码** | ~15 | VPU 解码，CPU≈0 |
| **第5轮** | **多线程流水线解耦** | **~40** | 检测不阻塞主循环 |
| **第6轮** | **GStreamer videoscale 硬件缩放** | **~40** | 省掉 Python resize+cvtColor |

### 9.2 当前性能（实测）

| 指标 | 数值 |
|------|------|
| 系统 FPS（推送频率） | ~40 |
| 行人检测（NPU） | ~50ms / 次 |
| 车牌识别（CPU HyperLPR3） | ~120ms / 次 |
| 视频解码 CPU 占用 | ~5%（VPU 硬解码） |
| WebSocket 带宽 | ~1KB/帧 |

### 9.3 RK3568 4核 CPU 分配

| 核心 | 任务 |
|------|------|
| 核0 | GStreamer 读帧线程 |
| 核1 | NPU 行人检测（每3帧） |
| 核2 | HyperLPR3 车牌识别（每9帧） |
| 核3 | 主推送循环 + 系统开销 |

## 11. 常见问题与排查

### 11.1 启动时警告（可忽略）

| 警告 | 含义 | 处理 |
|------|------|------|
| `rga_api version 1.9.3` | RGA 硬件加速库版本 | 正常日志 |
| `unable to query duration of stream` | RTSP 实时流无时长 | 正常 |
| `Cannot query video position` | RTSP 实时流无播放位置 | 正常 |
| `RKNN_ERR_MODEL_INVALID` (dynamic range) | 静态 shape 模型 | 正常，不影响推理 |
| `GPU device discovery failed` (ONNX Runtime) | RK3568 无独立 GPU | 正常，用 CPU 推理 |

### 11.2 NPU 相关

**Q: RKNN 模型加载失败**
```
检查：
1. .rknn 文件路径是否正确
2. librknnrt.so 版本是否与模型匹配（都是 2.3.2）
3. NPU 驱动是否正常: dmesg | grep rknpu
```

**Q: NPU 推理报维度错误**
```
原因：输入不是 4 维
解决：img = np.expand_dims(img, 0).astype(np.uint8)
确保 shape 为 (1, H, W, 3)
```

### 11.3 GStreamer 相关

**Q: GStreamer 硬解码不可用**
```
排查：
1. python3 -c "import cv2; print(cv2.getBuildInformation())" | grep GStreamer
   → 应显示 YES，否则需链接系统 OpenCV
2. gst-inspect-1.0 mppvideodec → 应有输出
3. gst-inspect-1.0 videoscale → 应有输出
4. 系统会自动回退到 FFmpeg 软解码
```

**Q: videoscale 输出分辨率错误**
```
确保 pipeline 中同时指定 width 和 height：
  videoscale ! video/x-raw,width=640,height=360 ! appsink

如果只指定 width，高度会保持原始值（如 640x1080）
```

### 11.4 环境相关

**Q: NumPy 版本不兼容**
```
系统 OpenCV 绑定 NumPy 1.x API，pip 安装的 NumPy 2.x 会报错：
  pip install "numpy<2"
```

**Q: ImportError: librknnrt.so**
```
sudo cp librknnrt.so /usr/lib/
sudo ldconfig
```

### 11.5 车牌识别相关

**Q: HyperLPR3 始终返回 0**
```
排查：
1. 确认画面中确实有车牌
2. 检查 ROI 区域是否正确
3. HyperLPR3 在小图上也能工作，不需要 1080p
```
